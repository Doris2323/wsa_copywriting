## 一、貼文結構：一開文 + 二串文

每則 Threads 貼文由三個部分組成，以「串文」形式連續發布：

### 【開文】吸引注意力 + 拋出核心觀點

功能：在 Threads 動態牆上搶佔注意力，讓人停下來看。
字數：150-300 字
結構：
1. 第一行：一句有力的斷言、反問句、或挑戰性觀點（這是整則貼文的生死線）
2. 展開痛點/現象描述（2-3 句，製造共鳴）
3. 帶出核心論點的方向預告
4. 結尾用「讓我來說說 👇」或「往下看 👇」引導串文

範例格式：
```
市面上 99% 的 Vibe Coding 實踐，都在犯同一個致命錯誤。

不是工具不好，不是 AI 不夠聰明——
是你根本沒給 AI 足夠精準的規格，就期待它寫出完美的程式。

就像叫一個新來的員工「把產品做好」，卻不給需求文件、不給驗收標準，然後怪他做出來的東西不對？

這不是 AI 的問題，是我們自己的問題。

但有一個方法，可以讓你的 Vibe Coding 正確率從 80% 拉到接近 100%——
讓我來說說 👇
```

### 【串文一】深度展開 + 論證核心觀點

功能：交付價值——給讀者一個帶得走的觀念或方法。
字數：200-400 字
結構：
1. 承接開文的論點，用一句話點明本串文的核心概念
2. 用比喻或案例具體說明（水球潘風格的生活化比喻是靈魂）
3. 帶入技術觀點或方法論（SDD、BDD、TDD 等）
4. 小結：用一句話濃縮本段核心，讓讀者有「啊，原來如此」的感覺

範例格式：
```
關鍵在於——你有沒有辦法寫出「讓 AI 一看就懂」的規格？

很多人的 Vibe Coding 是這樣的：
寫一段模糊的自然語言 Prompt → AI 產出一坨程式 → 人工 Review 找 Bug → 再下 Prompt 修修改改⋯⋯

這就像你告訴計程車司機「帶我去那個好吃的店」，然後抱怨他開錯路。

我們在 SDD（規格驅動開發）裡做的事完全不同：
我們先用 BDD 定義精確的「靶」——也就是可執行規格（Executable Specification），把每個功能的行為用 Given-When-Then 寫清楚。

然後才讓 AI 去「射箭」——去實作。

靶立好了，AI 射不中？沒關係，自動化測試會告訴它哪裡偏了，它會自己修正，直到正中紅心。

這才是負責任的 Vibe Coding 嘛！
```

### 【串文二】引發行動 + CTA 導流

功能：把觀點轉化為讀者的下一步行動。
字數：100-200 字
結構：
1. 呼應開文與串文一的核心觀點，用一句共鳴句收束
2. 根據貼文類型選擇適當的 CTA：
   - 知識型貼文 → 引導討論（「你覺得呢？留言聊聊」）
   - 社群導流 → 引導加入（Discord / Facebook 社群連結）
   - 課程相關 → 引導了解（課程頁面連結）
   - 活動宣傳 → 引導報名（活動連結 + 議程資訊）
3. 結尾語氣要溫暖有力，像朋友的邀請而非銷售推銷

範例格式：
```
如果你也受夠了「AI 寫的 code 要花更多時間 Debug」的無限迴圈——

歡迎來我們 SDD.tw 的 Discord 社群看看，每週二晚上九點，一群熱血工程師正在一起研究怎麼讓 AI 真正幫你寫出靠譜的程式。

不收費、不推銷，純粹一起研究、一起變強。

🔗 加入社群：waterballs.tw/submit-fbg

一起來吧 🙌
```

## 二、水球潘寫作風格規則（必遵守）

### 語氣與口吻
- 直接對讀者說「你」，製造一對一對話感
- 自信且帶權威感，敢於斷言：「絕對」「一定」「99%」
- 口語化但有深度，像技術演講的文字版
- 熱血但不雞湯，有態度但不說教
- 可以用「水球我⋯⋯」的第三人稱+第一人稱混用

### 修辭手法
- 每篇至少一個生活化比喻解釋技術概念（射箭與靶、老闆管下屬、拼拼圖等）
- 使用口語助詞：「嘛」「囉」「吼」「啊」
- 技術名詞第一次出現時附括號英文原文，例如：規格驅動開發（Specification-Driven Development, SDD）
- 短句開場，長句解釋；反問句推動論述
- 驚嘆號使用頻率高於一般寫作，可連用「！！」

### 句型武器庫（交替使用，避免重複）
- 斷言型：「在 AI 時代下最受重用的工程師，是最會 _____ 的工程師。」
- 反問型：「你覺得 _____ 最大的價值是什麼？」
- 批判型：「市面上 99% 的 _____ 都仍在錯誤的方向上。」
- 對比型：「不再 _____（痛點），正式 _____（好處）。」
- 號召型：「如果你和我一樣，_____ ，那麼你一定要 _____ 。」
- 金句型：「_____ 的勝利就是咱 _____ 的勝利。」

## 三、主題對齊規則（必遵守）

每則貼文必須歸屬於以下四大主題之一，並在撰寫時對齊其核心價值：

| 主題 | 核心一句話 | 適用場景 |
|------|---------|---------|
| ① WSA 社群凝聚力 | 臺灣最活躍的軟工社群 | 社群導流、活動宣傳、成果分享 |
| ② 軟工基礎 = AI 魔法基礎 | 沒有基礎，AI 再強也只是在猜 | 觀念教育、課程導購、反思型貼文 |
| ③ 工程師價值提升 | AI 來幫你升級，不是取代你 | 激勵型貼文、學員見證、價值倡導 |
| ④ 工程師轉型 | 能力左移，從寫 code 到定規格 | 趨勢分析、技能倡導、課程導購 |

### 主題植入方式
- 開文須自然呼應該主題的核心價值定位
- 串文一須融入該主題的核心論述
- 串文二的 CTA 須與該主題的文案方向一致
- 每則貼文僅聚焦 1 個主題，不混搭

## 四、關鍵字植入規則

### 從最新關鍵字報告中選取搭配關鍵字
- 每則貼文搭配 2-3 個熱門關鍵字，自然融入內文
- 優先選用討論熱度 ★★★★ 以上的關鍵字蹭熱度
- 至少包含一個與 SDD/BDD 相關的品牌關鍵字

### 標籤（Hashtag）使用規則
- 標籤不寫在貼文內文中（Threads 使用標籤功能，非內文 #）
- 每則貼文選擇 3-5 個標籤
- 標籤組合建議：1 個高熱度通用標籤 + 1 個中熱度技術標籤 + 1-2 個品牌標籤
- 品牌標籤固定使用：`sdd`、`AI x BDD 軟體全自動化開發術`
- 高熱度標籤範例：`vibe coding`、`ai coding`、`軟體工程師`
- 中熱度標籤範例：`Claude Code`、`Cursor`、`軟體工程師日常`

## 五、文案寫作標準檢核（必遵守）

### 排版規則
1. 數字清單的點號右側加一個空白：`1. 學員編號`
2. 中文標點一律全形：`，。！？（）`
3. 英文、中文和數字之間隔一個空白：`課後 10 個工作天`
4. 專有名詞首次出現：中文全名（英文全名, 英文簡寫），例如：`行為驅動開發（Behavior-Driven Development, BDD）`
5. 課程名稱加書名號：`《AI x BDD：規格驅動全自動開發術》`
6. 段落之間適當留白，保持社交距離

### 去除 AI 味檢核（極其重要）
撰寫完成後，逐條檢查並修正：

- [ ] 沒有 Emoji 轟炸（除關鍵 CTA 外不使用 Emoji）
- [ ] 沒有「這不是 A⋯⋯而是 B」的制式對比句型
- [ ] 沒有「老實說」「坦白說」「說真的」等制式開場白
- [ ] 沒有莫名堅定的數據（如「99% 的人都搞錯了」——除非是水球潘的標誌性風格且有脈絡支撐）
- [ ] 沒有說教感和雞湯感，有具體案例和個人經驗支撐
- [ ] 沒有過多「」引號強調非專有名詞
- [ ] 沒有不必要的括號補充說明
- [ ] 用詞符合台灣中文習慣（資料而非數據、程式設計而非編程）
- [ ] 文章的視覺節奏像真人寫作，非整齊的條列式

### 論述一致性檢核
- [ ] 本則貼文聚焦 1 個主軸觀點？
- [ ] 有足夠的例子或推論支撐觀點？
- [ ] 沒有中途引入無關主題或額外主張？
- [ ] 與品牌過去觀點無矛盾？
- [ ] 對應到「道／法／術／器」其中一項核心價值？

## 六、發文頻率與文體配比

### 三種文體輪替
| 文體 | 頻率 | 說明 |
|------|------|------|
| 乾貨＋資產知識文 | 每週 1 則 | 論述完整的技術觀點文，預期帶來可觀流量 |
| 信任＋影響力文 | 每週 2-3 則 | 像朋友聊天般分享靈感、觀察、趨勢觀點 |
| 信任＋友好文 | 每週 1-2 則 | Quote 誇獎他人貼文、分享認同他人論述 |

### 價值內容 vs 推廣內容比例
- 嚴格遵守 **4:1 原則**：每 4 則價值內容搭配 1 則推廣/導流內容
- 推廣貼文也須融入有價值的資訊，不能純廣告

## 七、規避事項（紅線）

以下為絕對不可觸碰的寫作紅線：

1. **不販賣焦慮**：不寫「不學就被淘汰」「再不跟上就來不及了」等恐懼訴求
2. **不豎立對立**：不寫「資深 vs 小白」「會的人 vs 不會的人」等鄙視鏈
3. **不空泛說教**：每個觀點必須有具體案例、比喻或數據支撐
4. **不過度推銷**：CTA 要像朋友的推薦，不像業務的話術
5. **不寫超出專業範圍的內容**：僅圍繞軟體工程方法論、AI 開發實踐、SDD 生態
6. **不散播未經驗證的資訊**：技術觀點須有根據，數據須可查證
